第3讲 serivce层代码的开发

在前面针对dao完成的四个范例代码中，我们可以发现，主函数main里的测试代码就功能上大体分为两种：一种负责数据的输入输出，还有一部分是负责录入数据的处理。沿用dao的开发思想，可以考虑是否能将main里的这些代码也按照功能分割提取，重新封装呢？答案当然是肯定的。

首先我们先来完成数据处理的这部分功能代码的封装。因为每次需要完成的业务不同，所以需要按不同的业务规则，给出针对性的数据验证，计算等功能，并将处理结果通过返回值，返回给其他程序流程。在编写这部分代码过程中，需要操作的业务数据可能来自两个方向，一个是通过参数传递进来的用户提供的实际业务数据；还有一部分是数据库里保存的原始业务数据，这些原始数据需要查询数据库获取，而最终处理好的数据也需要做数据持久化操作（永久保存或永久丢弃）。所有这些需要操作数据库的功能，我们在之前的dao程序里已经封装过，所以直接调用dao的方法即可。

    每一种业务需求都会有相应的处理代码，所以我们可以将完成某一功能的代码封装定义成一个函数。可以将相关业务函数分门别类的整合到不同的类里，这些类一般我们称之为“业务逻辑层“，通常命名为”***Service“。业务逻辑层（Business Logic Layer）无疑是系统架构中体现核心价值的部分。它的关注点主要集中在业务规则的制定、业务流程的实现等与业务需求有关的系统设计，也即是说它是与系统所应对的领域（Domain）逻辑有关，很多时候，也将业务逻辑层称为领域层。

    如果将这个开发过程比喻成去餐馆就餐的过程，那么service层就相当于是厨师。当我们到达餐馆要求就餐，我们就成为了用户，此时会有服务员出来询问点菜，并将点好的菜单传递给厨房的大厨，大厨只要接收来自于前面的菜单就好，并不实际负责招呼用户。确认了菜单里的菜品后，大厨也不需要去库房亲自挑选蔬菜，只需要通知采购人员，并获得所需蔬菜即可。此处采购员就承担了dao的角色，负责提供蔬菜（也就是原始数据）。当所需所有数据都准备好，大厨就可以开始制作（判断处理数据的代码）。最终菜品完成，只要将菜品返回即可，自然有人会将做好的菜呈现给用户。

第4讲 view层代码的开发

    下面我们再来完成数据输入输出代码的封装：这部分代码比较简单，主要完成数据的录入和结果的展示。它面向的群体是真正的用户，主要职责是接收用户的访问请求，并收集完成请求所需数据。当功能处理结束后，还需要向用户展示执行结果（当然这个结果可能是用户希望的，也可能是一些异常信息）。在餐馆就餐的例子中，每个餐厅都会有服务员，帮助用户点菜，点菜结束后，服务员得到足够的信息，并将这些信息通过参数传递给大厨。事实上，服务员不会亲自到厨房完成菜品的制作，这是厨师的工作。制作过程结束后，服务员又会站出来为我们上菜，这个过程其实就是在展示结果。当然如果制作过程出现问题，最终也是由服务员向我们解释错误原因。所以我们在定义这部分代码时，只需要收集数据传递给后面的程序，以及展示最终结果就好，不需要参与具体的数据处理过程。这部分程序我们一般称之为”显示逻辑“，命名时通常为"***View.java"。

第5讲 总结：java代码层的划分

    在开发一个应用时，根据代码所能完成的功能不同，我们可以将所编写的所有代码大致分为三部分：解决数据的来源；数据怎么处理；数据怎么显示。本着各司其职的思想，我们把软件设计成三个层次。这三个层次分别对应于三类需求。

    首先：是数据访问层。数据访问层是用来和数据打交道，具体的说，负责数据的增加，删除，修改和查询。数据访问层的对象（Data Access Object），简称 DAO。DAO 层对应着第一类需求：数据从哪儿来。

    其次：是业务逻辑层。业务逻辑层，是专门用来处理数据的，对应第二类需求。数据访问层也被称为service层。

    最后：用户提交的请求数据需要被接收，当数据处理完之后，结果数据需要显示给用户。而负责接收用户请求，并显示数据的是显示层，简称：view。

    把软件分成三个层次之后，典型的情况如下：view层与用户交互的过程中，接受用户的一个指令。如果 view 层用图形界面显示数据，则这个指令有可能是图形界面上的一次点击；如果 view 层用网页显示数据，则这个指令有可能是网页上发送的一个 http 请求。

当 view 层获得一个用户的指令之后，会把这个指令交给 service 层(调用service层中的方法)，把用户跟 view 层交互时输入的一些数据，传递给service 层。然后，等service层处理完成之后，把结果返回给 view 层，让 view 层显示。而service层在处理过程中，会调用 dao 层的方法。做数据的增删改查。因此，view、service、dao 三个层次之间，是从上到下依次调用方法的关系。示意图如下：

    上面我们介绍了软件的三层体系结构。那么，把软件设计成三层结构有什么好处呢？好处在于：当需求发生改变时，我们可以把改变局限在某个层次中，而不影响其他层次。例如，如果采购食材的地点以及放置物品的位置发生改变的话，我们不需要对前台服务员和厨师做过多的说明，只要让采购员能够清楚应该怎么工作就可以了。同样的，如果某一个层次的需求发生变化，则我们只需要针对那个特定的层次，修改相应的代码，而不用改变其他层次的代码。此外由于层是一种弱耦合结构，层与层之间的依赖是向下的，底层对于上层而言是“无知”的，改变上层的设计对于其调用的底层而言没有任何影响。所以在分层设计时，应遵循面向接口设计的思想，那么这种向下的依赖也就是一种弱依赖关系。在不改变接口定义的前提下，理想的分层式架构，应该是一个支持可抽取、可替换的“抽屉”式架构。在开发中为了体现三层体系结构，在项目打包过程中，我们会按照如下格式进行构建。

第6讲 事务控制

1.      转账引发的数据问题

    在上面的银行的例子中，service层的transfer()完成了转账的功能。但是这个方法真的能够使用而不出错吗？如果将代码改成下面这种情况会怎样呢？

 public void transfer(Integer fromId,String password,Integer toId,double money)

      throws Exception{

                     Account a=dao.getAccount(fromId);

                     if(a==null) throw new Exception("转账方卡号不存在!!");

                     if(!a.getPassword().equals(password)) throw new Exception("密码错误");

                     if(a.getBalance()<money) throw new Exception("余额不足");

                     Account b=dao.getAccount(toId);

                     if(b==null) throw new Exception("被转账方卡号不存在!!");

                     a.setBalance( a.getBalance() - money );

                     b.setBalance(b.getBalance()+money);

                     dao.updateAccount(a);

                     if(fromId==11)  throw new Exception("hehehe,出错了");

                     dao.updateAccount(b);

       }

我们考虑，如果fromId的值为11，那么转账动作就会在第二次更新数据库之前抛出异常，进而非正常结束方法的运行，不会再去执行dao.updateAccount(b)这句代码，这就意味着转账动作失败。根据现实情况，银行应该退回第一次扣除的金额，恢复数据库数据的原始状态。当我们运行后发现事实不是这样的，第一次更新数据库扣除的钱并没有因为第二次动作的失败而恢复，显然这不合理。出现这种现象的原因很简单，在oracle课程中，我们提出过，转账的两个更新动作是是一个事务，它们要么一起成功，要么一起失败。而在JDBC里，默认的事务提交策略是执行成功一条记录就提交一次。所以，上面的代码在异常发生时，第一次update动作的结果已经提交给数据库，没法改变了。

2.      Jdbc里的事务控制

银行的转账业务中需要将两次更新做成一个事务，而jdbc的默认事务提交策略无法完成这个功能，所以需要改变jdbc默认的自动提交策略，改为手动控制事务。JDBC中的事务手动控制需要使用的API：

1）Connection类的setAutoCommit(boolean ba)方法，参数ba如果是true表示自动提交，false表示手动提交。

   注意：此方法一经设置，永久生效。也就是说一旦设置conn.setAutoCommit(false),那么经由这个conn执行的命令的结果都需要手动提交，不需要反复设置。

2）Connection类的commit()和rollback()方法。完成事务的提交和回滚。

       手动控制事务后，transfer()方法的代码修改如下：

    public void transfer(Integer fromId,String password,Integer toId,double money)

      throws Exception{

              Connection conn =null;

              try{

                     conn=JdbcUtil4.getConnection();

                     conn.setAutoCommit(false);

                     Account a=dao.getAccount(fromId);

                     .......

                     a.setBalance( a.getBalance() - money );

                     b.setBalance(b.getBalance()+money);

                     dao.updateAccount(a);

                     if(fromId==11) throw new Exception("hehehe,出错了");

                     dao.updateAccount(b);

                     conn.commit();     //操作成功，没有出现任何异常，可以提交结果

              }catch(Exception e){

           conn.rollback();    //出现异常则进行回滚

           throw e;

              }finally{

           JdbcUtil4.close(null, null, conn);

              }

         } 

 

3.      ThreadLocal类

       经过改进，我们在代码里加入了事务控制的部分，但是整个程序协调运行时，能够达到理想效果吗？我们发现问题依然存在，事务控制并没有解决问题。分析service调用dao的过程，可以找到问题的根源：在转账方法内用到了数据库连接（用做事务控制），在dao层更新方法内也要用到数据库连接做update操作。这几个连接不是同一连接。我们对conn1调整提交策略，但是却用conn2操作数据库。根据事务的隔离性，对一个连接上事务的提交或回滚不会影响到另一连接上的事务的提交回滚。因此，上述实现存在的问题的本质是没有将两次更新操作放在同一事务中，这两次更新操作必须使用同一个连接对象。

1)       解决方法一：通过传递Connection对象解决问题

可以为dao方法添加Connection参数，将transfer里设置好提交策略的连接对象作为参数传递给dao，在dao里方法操作数据库时使用。当然这种方法缺点也是十分明显的，所有dao层接口和实现类的方法签名中都必须加入植入conn参数，丑陋且不利于接口的抽象。将dao的实现技术局限在Jdbc上。

2)       解决方法二：

修改JdbcUtils类中的getConnection方法，让每次返回的都是同一个数据库连接，dao层方法调用时都不能关闭此连接。这样只能用于单用户环境，如果是多个用户同时要使用此应用程序就会发生问题。例如：张三，李四共用一个SQLPlus窗口，跟JDBC中使用同一个数据库连接是类似的，张三执行的commit或rollback操作，会影响到李四执行的DML操作。

3)       解决方法三：通过ThreadLocal对象解决问题。

要达到这样一个目标，就能解决上述问题：多用户环境下，每个用户在执行一系列方法调用时，应当使用同一个连接对象。既不能是每次都使用新的连接对象，也不能大家共用同一个连接对象，应当是自己使用自己的连接对象。

现阶段程序所需的所有Connection都是通过JdbcUtil的getConnection()获取，那么我们可以控制getConnection()，使其在发放Connection对象时有一个鉴别判断的能力。当某个程序流程（线程，现在多用户环境下，每个用户肯定是用的自己的线程）调用getConnection()方法获取连接时，方法首先判断这个线程执行过程中是第几次调用方法获取连接。如果是第一次，那么生成一个新的连接返回给线程使用，如果不是第一次，则找到上一次线程获得的Connection对象返回，让其继续使用。这样，在一个程序流程中就只会出现一个线程对象，方便控制事务。

基于这个原理，摆在我们面前的障碍就是如何判断线程是第几次获取Connection对象。解决了这个问题，那么一切就都迎刃而解了。首先每一个线程对象被启动后，都会被分配一个独占私有的空间，用来存放线程的私有数据，变量类型可以是任意类型。我们可以在每个线程中存储一个私有的数据库连接，在这个线程之内，不论调用哪个方法，都使用此数据库连接。这块空间必须通过ThreadLocal来操作管理。它有两个方法get()取得当前线程中的某个私有变量，set(Object obj)存储某个对象至当前线程。当线程调用getConnection()方法获取连接时，我们可以先利用一个ThreadLocal对象去线程的空间里获取Connection值。如果没有找到，说明这是线程第一次来，那么我们为当前线程生成一个Connection对象，首先将其通过ThreadLocal对象存入线程的私有空间，然后再将Connection对象返回给线程运行使用。反之，如果通过ThreadLocal的一个对象在空间里获取到了Connection对象，则说明当前线程曾经获取过Connection，那么我们只要将这个Connection对象返回给线程继续使用即可。添加ThreadLocal对象后的代码如下：

    //声明一个静态的ThreadLocal变量，泛型表示它只能存取Connection类型的变量

private static ThreadLocal<Connection> tlo = new ThreadLocal<Connection>();     

// 获得与当前线程相关的数据库连接对象

public static Connection getConnection() throws Exception{

    // 获取当前线程中的私有Connection

              Connection conn = tlo.get();

       // 如果还没有，创建一个新的

              if(conn==null){

                Class.forName(map.getProperty("driver"));

                 conn = DriverManager.getConnection( map.getProperty("url"),

                     map.getProperty("username"),map.getProperty("password"));

          // 将新连接对象存储至当前线程

                 tl.set(conn);

              }

       // 如果当前线程中已经有创建好的连接对象，则直接返回

              return conn;

        }   

       接下来，在修改资源释放的代码，用于在关闭连接的基础上，清除当前线程存储的数据库连接。

    public static void close(ResultSet rs,Statement stm,Connection conn){

              if(rs!=null) try{rs.close();}catch(Exception e){}

              if(stm!=null) try{ stm.close(); }catch(Exception e){}

              if(conn!=null)

          try{

               conn.close(); 

               // 在连接关闭后设置ThreadLocal中Connection对象为null

                            //以便在接下来的方法调用中使用新的连接

               tl.remove();

          }catch(Exception e){}

       }    

 

    通过上面一系列的修改后，线程在完成一个事务过程中只会得到一个Connection对象。作为事务的一个动作（dao方法）在执行结束后，就不能关闭Connection，否则会直接导致事务的后续动作无法执行，。那么连接应该在什么位置关闭呢？原则上Connection是分配给线程用来完成事务的，所以应在线程结束的时候关闭。但是线程的结束无法准确判断出来，那么我们的底线是至少保证在一个事务内使用一个连接，在事务开始时控制事务的提交策略，事务结束连接也随之关闭。
